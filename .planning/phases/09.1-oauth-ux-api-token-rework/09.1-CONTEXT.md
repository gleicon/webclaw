# Phase 9.1: OAuth UX & API Token Rework - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix broken OAuth integrations and improve auth UX for all four social providers. GitHub and Notion OAuth flows require a `client_secret` in the token exchange — something a pure browser app cannot hold — so they are replaced with Personal Access Token (PAT) and Internal Integration Token inputs respectively. Twitter and Google PKCE stays intact but the setup UX (Client ID input, redirect URI guidance) needs to be prominent and clear.

What is NOT in scope:
- Adding new integrations (Spotify, Slack, Discord)
- Server-side token exchange proxy
- Changes to tool logic beyond token retrieval

</domain>

<decisions>
## Implementation Decisions

### GitHub Auth Replacement
- Replace GitHub OAuth with **Personal Access Token (PAT) only** — no OAuth app needed
- UI shows required scopes inline next to the input field: `repo, read:user`
- Include a direct link: `github.com/settings/tokens/new?scopes=repo,read:user`
- When a PAT is saved, it **silently replaces** any existing GitHub OAuth token — no disconnect step required
- No hybrid mode: OAuth path is removed for GitHub entirely

### Notion Auth Replacement
- Replace Notion OAuth with **Internal Integration Token** input
- UI shows inline hint: "Create an integration at developers.notion.com" with a direct link
- Note that users must share pages with the integration (callout, not a full walkthrough)
- Same silent-replace behavior as GitHub

### Token Storage
- GitHub PAT and Notion Integration Token stored in the **existing encrypted token_store** (IndexedDB, AES-256-GCM)
- Stored under provider key ('github', 'notion') with an `auth_type: "pat"` flag to distinguish from OAuth tokens
- `GetToken(provider)` returns the PAT the same way it returns OAuth tokens — **zero changes to tool code**
- `auth_type: "pat"` signals: skip refresh logic (PATs don't expire on a refresh cycle)
- UI **masks token after save** — shows `••••••••••` with show/hide toggle and "Update token" button, matching the API key masking pattern already used in Settings

### Client ID Storage (Twitter / Google)
- Client IDs remain in **localStorage** under `webclaw_oauth_clientids` — they are public values (appear in OAuth redirect URLs), not secrets
- No change to the existing `webclaw.oauth.setClientId()` persistence mechanism

### Twitter / Google OAuth Setup UX
- Each OAuth provider card (Twitter, Google) has a **Client ID text field** directly on the card, above the Connect button — visible at a glance, no extra modal
- An **inline callout box** shows the required redirect URI with a copy button:
  ```
  Set your OAuth App's callback URL to: about:blank  [Copy]
  ```
  This callout is permanently visible on the card (not hidden behind a tooltip)
- Callout appears before the Connect button so users see it before attempting to connect

### Connected Services UI Differentiation
- Each provider card shows a **small auth-type badge**:
  - Twitter, Google: badge labeled "OAuth 2.0"
  - GitHub, Notion: badge labeled "Access Token"
- Connection status indicator: **green dot + "Connected" text** when active; gray dot + "Not connected" when not
- Invalid token state: **red dot + "Invalid token"** inline on the card, plus a link to re-enter the token
- Tools also return user-friendly errors on invalid token: e.g., "GitHub token is invalid. Please update it in Settings."

### Auth Type Routing
- `OAuthManager.GetToken(provider)` is the single retrieval path for all auth types
- PAT-backed providers skip the token refresh goroutine (detected via `auth_type` flag in stored token)
- `OAuthManager.IsConnected(provider)` returns `true` if a non-empty token exists (OAuth or PAT)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `internal/oauth/token_store.go`: `SaveToken()` / `GetToken()` — reuse directly for PAT storage by extending the stored token struct with an `AuthType` field ("oauth" | "pat")
- `internal/oauth/manager.go`: `IsConnected()`, `Disconnect()`, `ListConnections()` — all remain valid for PAT-backed providers
- `internal/oauth/js_exports.go`: `setClientId`, `isConnected`, `getConnectionStatus` exports — no changes needed
- `internal/oauth/providers.go`: `OAuthProvider` struct — GitHub/Notion entries remain in registry for display purposes but `InitiateConnection()` branches on `auth_type`
- `index.html` Connected Services section: card rendering + connect/disconnect wiring — extend cards, don't replace

### Established Patterns
- API key masking in Settings tab: mask-after-save with show/hide toggle — replicate for PAT fields
- `webclaw.oauth.setClientId(provider, id)` called from JS on every WASM load — Client ID persistence pattern unchanged
- Tool graceful-failure messages ("Please connect X in Settings first") — extend to cover invalid-token case

### Integration Points
- `cmd/webclaw/main.go` OAuth goroutine: `InitiateConnection()` is the only call site that needs branching (OAuth popup vs. PAT save)
- `index.html` Settings > Connected Services: card template needs auth-type badge, Client ID field (OAuth), Token field (PAT), and inline callout
- `internal/oauth/manager.go:InitiateConnection()`: add early return / alternate path for PAT providers

</code_context>

<specifics>
## Specific Ideas

- The PAT token field should behave identically to the API key input in the existing Settings panel: password input type, masked after save, show/hide toggle, update button
- The redirect URI callout should use the same callout/alert style already present in the UI — consistent visual language
- Auth-type badges should be small and subtle (e.g., `text-xs` Tailwind chip in the card header area) — informative, not dominant
- "Invalid token" state should be detectable by tools when they receive a 401/403 from the API; tools should call back to mark the provider as invalid rather than just returning an error string

</specifics>

<deferred>
## Deferred Ideas

- Server-side PKCE proxy for GitHub/Notion OAuth — would require a server component, breaks zero-server constraint
- Spotify integration — separate phase
- Fine-grained PAT token scope validation (verifying the PAT has the required scopes before saving) — could be a follow-up UX improvement
- GitHub Device Flow as an alternative to PAT — lower priority now that PAT path is chosen

</deferred>

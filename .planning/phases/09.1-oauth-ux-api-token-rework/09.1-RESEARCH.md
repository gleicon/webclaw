## Summary

Phase 9.1 is a surgical rework of the Connected Services authentication layer. The root cause driving this phase is that GitHub and Notion OAuth token exchange requires a `client_secret`, which a pure browser app cannot hold securely (it would be visible in source). The PKCE flow used by Twitter and Google does not require a `client_secret`, so those flows remain intact.

The work splits cleanly into two tracks. Track A is Go/WASM: extend the `Token` struct with `AuthType string`, add a `SavePAT(provider, pat)` method on `OAuthManager` that constructs a Token with `auth_type: "pat"` and zero `ExpiresAt` (never expires), add an early-return branch in `InitiateConnection` for PAT providers, and update `GetToken` to skip refresh logic when `AuthType == "pat"`. Track B is `index.html` JS/HTML/CSS: replace the GitHub and Notion card content (swap out the Connect button and Client ID field for a password-style PAT input plus save/show-hide/update UX), add auth-type badges to all four provider cards, add a visible redirect-URI callout to the Twitter and Google cards, and implement the red-dot invalid-token state.

The existing encrypted token store, `SaveToken`/`LoadToken`, `IsConnected`, `Disconnect`, and `GetConnectionStatus` are all reusable without modification to their core logic. The `AuthType` field addition to the `Token` struct is backward-compatible because the field is omitempty-compatible in JSON — existing stored tokens will deserialize with `AuthType == ""`, which is treated as OAuth.

**Primary recommendation:** Extend `Token` with `AuthType`; branch `InitiateConnection` and `GetToken`; rework the two PAT provider cards in index.html; add badges and callouts to all four cards.

## Architecture Patterns

### Go/WASM: Token Struct Extension

The `Token` struct in `token_store.go` currently has no `AuthType` field. Adding one is non-breaking:

```go
// In internal/oauth/token_store.go
type Token struct {
    Provider     string    `json:"provider"`
    AccessToken  string    `json:"access_token"`
    RefreshToken string    `json:"refresh_token"`
    ExpiresAt    time.Time `json:"expires_at"`
    Scope        string    `json:"scope"`
    AuthType     string    `json:"auth_type,omitempty"` // "oauth" | "pat" — blank means oauth
    Username     string    `json:"username,omitempty"`
    UserID       string    `json:"user_id,omitempty"`
}
```

Existing stored OAuth tokens will deserialize with `AuthType == ""`. Any code checking for PAT must check `token.AuthType == "pat"`, not `token.AuthType != "oauth"`. This handles existing-token migration cleanly.

### Go/WASM: SavePAT Method on OAuthManager

A new `SavePAT(provider, pat string) error` method on `OAuthManager` constructs a `Token` with:
- `AuthType: "pat"`
- `AccessToken: pat`
- `RefreshToken: ""` (empty — no refresh)
- `ExpiresAt: time.Time{}` (zero value — never expires)

Then calls the existing `tokenStore.SaveToken(provider, token)`. This is a thin wrapper — no new IndexedDB logic needed.

### Go/WASM: InitiateConnection Guard for PAT Providers

`OAuthManager.InitiateConnection` currently does the full popup OAuth flow. The JS layer will not call it for PAT providers (Connect button is replaced by PAT input), but a defensive guard prevents accidents:

```go
func (m *OAuthManager) InitiateConnection(providerName string) error {
    if providerName == "github" || providerName == "notion" {
        return fmt.Errorf("%s uses a Personal Access Token — use savePATToken instead", providerName)
    }
    // ... existing OAuth popup flow unchanged
}
```

### Go/WASM: GetToken — PAT Already Safe

`OAuthManager.GetToken` calls `token.NeedsRefresh()` which returns `false` when `RefreshToken == ""`. Because PATs will have an empty `RefreshToken`, the existing logic already skips refresh for PATs.

`IsExpired()` also returns `false` when `ExpiresAt.IsZero()`. PATs will have zero `ExpiresAt`. So PATs flow through `GetToken` correctly with the existing code.

Adding an explicit early return guards against future regressions:

```go
func (m *OAuthManager) GetToken(providerName string) (string, error) {
    token, err := m.tokenStore.LoadToken(providerName)
    // ... nil/error checks unchanged ...

    // PAT tokens: return immediately, skip refresh logic
    if token.AuthType == "pat" {
        if token.AccessToken == "" {
            return "", fmt.Errorf("please update your %s token in Settings", providerName)
        }
        return token.AccessToken, nil
    }

    // ... existing OAuth refresh logic continues ...
}
```

### Go/WASM: New JS Export — savePATToken

Added in `js_exports.go` following the exact pattern of the existing `initiateConnection` export:

```go
savePATFn := js.FuncOf(func(this js.Value, args []js.Value) interface{} {
    if len(args) < 2 {
        return js.Undefined()
    }
    provider := args[0].String()
    pat := args[1].String()

    promiseCtor := js.Global().Get("Promise")
    return promiseCtor.New(js.FuncOf(func(this js.Value, resolveReject []js.Value) interface{} {
        resolve := resolveReject[0]
        reject := resolveReject[1]
        go func() {
            err := m.SavePAT(provider, pat)
            if err != nil {
                reject.Invoke(err.Error())
                return
            }
            resolve.Invoke(js.Undefined())
        }()
        return nil
    }))
})
oauthObj.Set("savePATToken", savePATFn)
```

### JS: Invalid Token State

Add a JS-side `invalidProviders` Set and a new exported function `webclaw.oauth.markInvalid(provider)` backed by an in-memory `map[string]bool` on `OAuthManager`. `GetConnectionStatus` reads this map and includes a `valid: false` field in returned status JSON. The Connected Services UI checks this field when rendering the red-dot state.

On the Go side:

```go
type OAuthManager struct {
    // ... existing fields
    invalidTokens map[string]bool // transient; resets on page reload
}

func (m *OAuthManager) MarkInvalid(provider string) {
    m.invalidTokens[provider] = true
}
```

`GetConnectionStatus` adds `Valid: !m.invalidTokens[providerName]` to the returned `ConnectionStatus` struct. Tools call `webclaw.oauth.markInvalid(providerId)` on 401/403 HTTP responses.

### Anti-Patterns to Avoid

- **DOM string injection with variable content**: CLAUDE.md and the existing security hook block dynamic innerHTML with variable content. All new card elements must use `createElement` + `textContent` + `appendChild`. The existing `disconnectModal` template literal at line 1741 uses only a trusted provider display name — acceptable. New PAT/token content MUST use DOM methods.
- **Modifying tool code for token retrieval**: `GetToken(provider)` must remain the single retrieval path. Tool code is not touched.
- **Storing PAT in localStorage**: PATs are secrets — they must go through the existing encrypted token_store (IndexedDB, AES-256-GCM), not localStorage. Client IDs (public values) stay in localStorage.
- **Calling InitiateConnection for GitHub/Notion**: The JS layer must not call `webclaw.oauth.initiateConnection("github")` — only `webclaw.oauth.savePATToken("github", pat)`.

## Common Pitfalls

### Pitfall 1: Backward-Compatibility of Token Deserialization

**What goes wrong:** Adding `AuthType` to `Token` struct could break deserialization of existing tokens if code treats empty `AuthType` as an error.
**Why it happens:** Existing stored tokens have no `auth_type` key in their JSON. `json.Unmarshal` will leave the field as zero-value (empty string).
**How to avoid:** Use `json:"auth_type,omitempty"` and always check `token.AuthType == "pat"`, never `token.AuthType == "oauth"`. Empty string is the OAuth default.
**Warning signs:** After page reload, `GetToken` triggers a refresh attempt for a PAT token.

### Pitfall 2: NeedsRefresh False Positives for PATs

**What goes wrong:** If a PAT somehow gets stored with a non-zero `ExpiresAt` (e.g., a future code change accidentally sets it), `NeedsRefresh` will return true and fail because `RefreshToken == ""`.
**Why it happens:** `NeedsRefresh` checks `ExpiresAt` before checking `RefreshToken`.
**How to avoid:** `SavePAT` must explicitly set `ExpiresAt: time.Time{}` (zero value) and `RefreshToken: ""`. The explicit early-return in `GetToken` for `AuthType == "pat"` provides a second layer of defense.

### Pitfall 3: IndexedDB Version Not Needed

**What goes wrong:** A developer might think schema changes require bumping the IndexedDB version number.
**Why it happens:** The `Token` struct's `AuthType` field is transparent to IndexedDB — it's serialized inside the encrypted ciphertext blob. The `oauth` object store schema (key: provider string, value: StoredToken with ciphertext/iv/salt/created_at) does not change.
**How to avoid:** Do NOT bump the IndexedDB version. Schema is unchanged.
**Warning signs:** `"database blocked - close other tabs"` error in console after a version bump.

### Pitfall 4: DOM Security Hook on PAT Field

**What goes wrong:** The security hook blocks dynamic content injection. If a code path tries to set `statusSpan.textContent` with a raw token value, or builds a string containing the PAT, the hook fires.
**Why it happens:** Developer habit of using string interpolation for status messages.
**How to avoid:** The masked display shows a fixed string (e.g., "Connected") or uses `input.type = "password"`. Never put the actual token value in DOM text or as an element attribute.

### Pitfall 5: WASM Goroutine Timing for savePATToken

**What goes wrong:** `savePATToken` is registered inside the OAuth goroutine in `main.go` which runs 300ms after startup. If JS calls it before the goroutine completes, the function won't exist.
**Why it happens:** The OAuth goroutine has a `time.Sleep(300 * time.Millisecond)` delay (main.go line 221).
**How to avoid:** The JS-side PAT save is only triggered by a user clicking "Save" in Settings — which requires the UI to be built after WASM is ready. The existing `if (!window.webclaw || !window.webclaw.oauth)` guard pattern handles this. No special timing logic is needed beyond what already exists.

### Pitfall 6: markInvalid State Resets on Reload

**What goes wrong:** If `invalidProviders` is stored in-memory on `OAuthManager`, it resets on page reload. The red-dot indicator disappears even though the token is still invalid.
**Why it happens:** By design — only tools discover invalid tokens at runtime.
**How to avoid:** This is acceptable behavior per the CONTEXT.md ("invalid token state should be detectable by tools when they receive a 401/403"). The user will see the error again on next tool call. If persistence is needed, it can be stored in a separate IndexedDB key — but this is deferred per CONTEXT.md.

**Why GitHub OAuth cannot work in a pure browser app (HIGH confidence):**
GitHub's token exchange endpoint (`https://github.com/login/oauth/access_token`) requires `client_secret` in the POST body. GitHub does not support PKCE — it uses the authorization code flow designed for confidential clients (server-side apps). A browser app has no way to keep a client secret confidential. This cannot be worked around without a server-side token exchange proxy.

**Why Notion OAuth cannot work in a pure browser app (HIGH confidence):**
Notion's token endpoint (`https://api.notion.com/v1/oauth/token`) requires HTTP Basic Authentication with the client secret in the Authorization header. This is the confidential client pattern. No PKCE support is documented in Notion's OAuth spec.

   - What we know: CONTEXT.md says show "a callout, not a full walkthrough" about sharing pages with the integration.
   - What's unclear: Exact Tailwind styling.

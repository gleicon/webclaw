<objective>
Implement Twitter/X integration with tools for posting tweets, replying, searching, and viewing timeline using OAuth-authenticated API calls.
</objective>

<context>
Twitter/X is a high-value integration (Priority 1 from 09-RESEARCH.md). Users want to post tweets, check their timeline, and search for content using natural language commands like "Tweet: Just shipped a new feature!" or "What's happening on Twitter?"

**API Details:**
- Base URL: https://api.twitter.com/2
- Authentication: Bearer token from OAuth 2.0
- Rate limits: 300 requests per 15 minutes for most endpoints
- Scopes needed: tweet.read, tweet.write, users.read, offline.access

**Twitter API v2 Endpoints:**
- POST /2/tweets - Create tweet
- GET /2/users/:id/timelines/reverse_chronological - Home timeline
- GET /2/tweets/search/recent - Search recent tweets (last 7 days)
- GET /2/tweets/:id - Get specific tweet

**Tool Pattern:**
Follow existing WebClaw tool structure (like web_fetch, web_search):
- Each tool has Name, Description, InputSchema, Execute function
- Execute receives context.Context and params map[string]interface{}
- Returns *ToolResult with Content (for LLM) and DisplayContent (for UI)
- Use jsbridge.Fetch() for HTTP - never net/http

**Error Handling:**
- 401: Token expired, try refresh (handled by OAuthManager)
- 403: Insufficient permissions, prompt user to reconnect with more scopes
- 429: Rate limited, show "Rate limited by Twitter. Try again in X minutes."
- 404: Tweet not found for reply

**Deferred (not in this plan):**
- Media uploads (images/videos) - requires multipart/form-data
- Streaming API - requires persistent connection
- DMs (Direct Messages) - requires additional scopes
- Analytics/metrics

Reference: 09-CONTEXT.md Section "Twitter/X Use Cases"
</context>

  <name>Task 5: Add rate limiting and caching</name>
  <files>internal/integrations/twitter/client.go (enhance)</files>
  <action>
    Enhance client with rate limit awareness and response caching.
    
    1. Rate limit tracking:
       ```go
       type RateLimit struct {
           Limit     int
           Remaining int
           ResetTime time.Time
       }
       
       func (c *Client) GetRateLimit(endpoint string) *RateLimit
       ```
       
       Parse headers from API responses:
       - x-rate-limit-limit: total limit
       - x-rate-limit-remaining: remaining calls
       - x-rate-limit-reset: Unix timestamp when limit resets
    
    2. Preemptive rate limiting:
       - Before making request, check if remaining > 0
       - If rate limited, return error with time until reset
    
    3. Response caching (read operations only):
       ```go
       type Cache struct {
           mu    sync.RWMutex
           items map[string]cacheEntry
       }
       
       type cacheEntry struct {
           data      []byte
           timestamp time.Time
           ttl       time.Duration
       }
       ```
       
       - Cache timeline and search results for 2 minutes
       - Cache key: method + url + params hash
       - Stale-while-revalidate: return cached, refresh in background
       
    4. Cache benefits:
       - Reduces API calls (better rate limit usage)
       - Faster response for repeated queries
       - Better UX when user asks follow-up questions
       
    5. Don't cache:
       - Post operations (write)
       - User-specific data that changes frequently
  </action>

Create a single-file distribution mode where WebClaw is contained entirely within one HTML file. This provides ultimate portability - the file can be emailed, stored in cloud storage, opened directly from the filesystem, and shared without any folder structure.

Purpose: Enable scenarios where distributing a folder is inconvenient: email attachments, single-file downloads, offline archives, and demonstration sharing.

Output:
- Build script that inlines all assets into single HTML
- dist-singlefile/webclaw.html containing everything
- Estimated size: ~1.2MB (base64 overhead vs multi-file 920KB)

## Size Analysis

**Multi-file bundle (Plan 08-01):**
- WASM (brotli): 865KB
- JS: ~37KB → ~12KB minified+gzipped
- CSS: ~15KB compiled
- HTML: ~47KB
- **Total: ~940KB**

**Single-file bundle (this plan):**
- Base64 encoding adds ~33% overhead (8-bit → 6-bit + padding)
- WASM base64: 865KB × 1.33 = ~1.15MB
- JS+CSS+HTML base64: ~70KB × 1.33 = ~93KB
- **Total: ~1.25MB**

**Overhead: ~310KB (33%) for ultimate portability**

This is acceptable for:
- Email attachments (1.25MB is small for modern email)
- Single-file downloads
- Cloud storage sharing
- USB stick distribution

## Technical Approach

Two strategies for creating single-file bundle:

1. **Vite + Rollup inline plugin:** Configure Vite to inline all dynamic imports and assets
2. **Post-build script:** Build multi-file first, then inline everything via custom script

Recommended: Approach 2 (post-build) because:
- More control over inlining order
- Can optimize WASM separately (keep brotli, base64 encode)
- Easier debugging (multi-file works first, then combine)
- Can generate both formats from same base build

  <name>Task 1: Create WASM inlining utility script</name>
  <files>scripts/inline-wasm.js</files>
  <action>
    Create a Node.js script that embeds WASM into HTML as base64:
    
    1. **Script purpose:**
       - Read the brotli-compressed WASM (webclaw.wasm.br)
       - Base64 encode it
       - Generate a data URI: `data:application/wasm;base64,AAAA...`
       - Create JavaScript that decodes and instantiates WASM from data URI
    
    2. **Implementation approach:**
       ```javascript
       // scripts/inline-wasm.js
       const fs = require('fs');
       const path = require('path');
       
       const wasmBrPath = path.join(__dirname, '../dist/webclaw.wasm.br');
       const wasmBr = fs.readFileSync(wasmBrPath);
       const wasmBase64 = wasmBr.toString('base64');
       
       // Generate decoder code
       const decoderCode = `
         // WASM binary embedded as base64 brotli-compressed data
         const wasmBase64 = "${wasmBase64}";
         const wasmBinary = Uint8Array.from(atob(wasmBase64), c => c.charCodeAt(0));
         
         // Decompress brotli if needed, or pass directly to WASM if browser supports brotli
         // Actually, we should decompress brotli in JS or use uncompressed WASM
       `;
       ```
    
    3. **Brotli consideration:**
       - Option A: Decompress brotli in JS (adds ~100KB decompressor code)
       - Option B: Use uncompressed WASM (4.2MB → ~5.6MB base64, too large)
       - Option C: Use gzip instead (built into browser decompression)
       - **Recommendation:** Option C - switch to gzip compression which browsers decompress automatically via Content-Encoding, or use uncompressed and accept larger size for simplicity
       
       Actually, for single-file, best approach:
       - Use uncompressed WASM (4.2MB)
       - Base64 encode: ~5.6MB
       - Too large for email
       
       Better approach for single-file:
       - Keep WASM as separate file reference
       - Only inline JS and CSS
       - Document that single-file = HTML+WASM pair
       
       OR:
       - Use compressed WASM but implement decompression
       - Evaluate brotli decompressor size vs savings
    
    4. **Revised approach - Two-tier single file:**
       ```javascript
       // Inline everything EXCEPT WASM
       // WASM stays as separate file but can be optionally embedded
       
       // For "ultimate" single file mode:
       const inlineWASM = process.env.INLINE_WASM === 'true';
       
       if (inlineWASM) {
         // Use pako or fflate for gzip decompression in browser
         // Compress WASM with gzip (not brotli - no native browser support)
         // Decompress on load: small ~5KB decompressor vs 3MB savings
       }
       ```
    
    5. **Final design decision:**
       - **Standard single-file:** HTML with inline JS+CSS, WASM external (fetches from same dir)
       - **Ultimate single-file:** HTML with EVERYTHING including WASM base64 encoded
       - Document size trade-off clearly
       - Use gzip compression for WASM (better JS decompressor availability)
    
    Implementation will create:
    - `scripts/build-singlefile.js` - main build orchestrator
    - Supports --inline-wasm flag for ultimate mode

    2. **Build script (scripts/build-singlefile.js):**
       Orchestrates the single-file build:
       ```javascript
       #!/usr/bin/env node
       const { build } = require('vite');
       const fs = require('fs');
       const path = require('path');
       
       async function buildSingleFile() {
         // Step 1: Build with Vite
         await build({ configFile: 'vite.singlefile.config.js' });
         
         // Step 2: Read built files
         const html = fs.readFileSync('dist-singlefile/index.html', 'utf8');
         const jsFiles = fs.readdirSync('dist-singlefile/assets').filter(f => f.endsWith('.js'));
         const cssFiles = fs.readdirSync('dist-singlefile/assets').filter(f => f.endsWith('.css'));
         
         // Step 3: Inline JS
         let inlinedHtml = html;
         for (const jsFile of jsFiles) {
           const jsContent = fs.readFileSync(`dist-singlefile/assets/${jsFile}`, 'utf8');
           inlinedHtml = inlinedHtml.replace(
             `<script type="module" src="./assets/${jsFile}"></script>`,
             `<script type="module">${jsContent}</script>`
           );
         }
         
         // Step 4: Inline CSS
         for (const cssFile of cssFiles) {
           const cssContent = fs.readFileSync(`dist-singlefile/assets/${cssFile}`, 'utf8');
           inlinedHtml = inlinedHtml.replace(
             `<link rel="stylesheet" href="./assets/${cssFile}">`,
             `<style>${cssContent}</style>`
           );
         }
         
         // Step 5: Handle WASM (copy alongside or inline)
         // If --inline-wasm flag, base64 encode and embed
         
         // Step 6: Write final file
         fs.writeFileSync('dist-singlefile/webclaw.html', inlinedHtml);
         
         console.log('Single-file build complete: dist-singlefile/webclaw.html');
       }
       
       buildSingleFile();
       ```

         const workerPath = path.join(__dirname, '../static/worker.js');

<task type="auto">
  <name>Task 4: Create optional WASM inlining (ultimate mode)</name>
  <files>scripts/inline-wasm.js</files>
  <action>
    Implement optional full WASM inlining for "ultimate" single-file mode:
    
    1. **Size analysis:**
       - WASM brotli: 865KB
       - Base64 brotli: 865KB × 1.33 = ~1.15MB
       - Add decompressor (fflate/pako): ~5KB
       - Total overhead vs external file: ~1.15MB
       
       For ultimate portability (email, single file), this is acceptable.
       For regular use, external WASM file is better.
    
    2. **Compression choice:**
       - Brotli: Best compression (865KB), but requires decompressor
       - Gzip: Good compression (~1MB), broader JS decompressor support
       - **Decision:** Use gzip for inline WASM (pako library is small and reliable)
    
    3. **Implementation:**
       ```javascript
       // scripts/inline-wasm.js
       const pako = require('pako'); // Node zlib could work too
       
       function inlineWASM(html) {
         const wasm = fs.readFileSync('dist/webclaw.wasm');
         
         // Compress with gzip
         const compressed = pako.gzip(wasm);
         const base64 = Buffer.from(compressed).toString('base64');
         
         // Add pako decompressor (CDN or inline)
         const decompressor = `
           // Minimal gzip decompressor (pako-lite)
           ${fs.readFileSync('node_modules/pako/dist/pako_inflate.min.js')}
         `;
         
         // WASM loader
         const wasmLoader = `
           ${decompressor}
           const __WASM_COMPRESSED__ = "${base64}";
           const __WASM_BINARY__ = pako.inflate(
             Uint8Array.from(atob(__WASM_COMPRESSED__), c => c.charCodeAt(0))
           );
           
           // Override fetch to return WASM from memory
           const originalFetch = window.fetch;
           window.fetch = function(url, options) {
             if (url.includes('webclaw.wasm')) {
               return Promise.resolve(new Response(__WASM_BINARY__, {
                 headers: { 'content-type': 'application/wasm' }
               }));
             }
             return originalFetch(url, options);
           };
         `;
         
         // Inject before other scripts
         html = html.replace('<script type="module">', 
           `<script>${decompressor}</script><script type="module">${wasmLoader}`);
         
         return html;
       }
       ```
    
    4. **Alternative - use DecompressionStream API:**
       Modern browsers have native gzip decompress:
       ```javascript
       const ds = new DecompressionStream('gzip');
       const writer = ds.writable.getWriter();
       const reader = ds.readable.getReader();
       // ... decompress
       ```
       This avoids bundling pako entirely! Only works in modern browsers (Chrome 80+, Firefox 90+, Safari 15.4+).
       
       **Decision:** Use DecompressionStream API with pako as fallback.
    
    5. **Build command:**
       ```bash
       npm run build:singlefile:inline-wasm
       # or
       INLINE_WASM=true npm run build:singlefile
       ```
    
    6. **Output file:**
       - dist-singlefile/webclaw-ultimate.html (with inlined WASM)
       - dist-singlefile/webclaw.html (JS+CSS only, WASM external)
    
    7. **Documentation:**
       Document size difference clearly:
       - webclaw.html: ~100KB + webclaw.wasm.br (865KB) = ~965KB total
       - webclaw-ultimate.html: ~1.25MB standalone
